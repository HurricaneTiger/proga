# Minecraft LAN Tunnel MVP

A monorepo implementing a relay-based tunnel that lets Minecraft LAN games reach players outside the local network. A host connects to a relay server via WebSocket and creates a room; a client joins via an invite code. TCP traffic between the client's Minecraft and the host's Minecraft server is piped bidirectionally through the WebSocket relay. The Electron desktop app provides a dark-themed React UI in Russian. An integration test proves the data path works end-to-end using JSON-based tunnel messages.

**Watch for:** The desktop app sends binary-framed tunnel data (via `encodeFrame`) but the relay server only parses JSON messages — these two paths are incompatible in production (confirmed). The relay broadcasts host-to-client tunnel data to *all* clients in the room rather than routing by session ID (confirmed). The invite code is generated with `Math.random`, which is predictable enough to allow brute-force room takeover in a public deployment (confirmed). No authentication, rate-limiting, or TLS configuration exists on the relay server (confirmed).

---

## High-level view

The workspace is cleanly layered: `packages/shared` defines the protocol types and constants, `packages/tunnel-core` provides binary frame encoding and exponential-backoff reconnection, `apps/relay-server` manages rooms and relays messages, and `apps/desktop` wraps the Electron UI and tunnel logic. The separation is sensible — shared types prevent drift and tunnel-core is reusable outside Electron.

The protocol has a fundamental split personality. The relay server speaks exclusively JSON (parsing every incoming WebSocket message via `JSON.parse`), while the desktop host and client encode tunnel payloads as compact binary frames using `tunnel-core`. These two encoding schemes never meet: the Electron app would fail at runtime because the relay drops or errors on binary messages. The integration test sidesteps this by manually performing the relay logic with JSON messages, never exercising the desktop app's binary path through the actual relay.

Room management is simple and correct for an MVP. Room capacity is bounded, expired rooms are reaped, disconnects are propagated. The host-to-client data forwarding has a fan-out flaw: the relay broadcasts tunnel data to every connected client regardless of which session originated the data, which means one player's packets arrive at all players.

Security posture is stub-level. No authentication on room creation or joining, 6-character alphanumeric invite codes with `Math.random` entropy (36^6 ~ 2.2 billion combinations, but easily brute-forced at WebSocket speed), no TLS termination story, and the relay trusts all incoming messages without size or rate limits.

<details>
<summary>Issues (7)</summary>

1. **Binary/JSON protocol mismatch** — The desktop app encodes tunnel data as binary frames (`encodeFrame`), but the relay server only calls `JSON.parse(data.toString())`. Binary messages will throw parse errors and be dropped. Either the relay needs a binary handler or the desktop app should send JSON tunnel messages matching the relay's expectations.

2. **Host-to-client broadcast instead of routing** — `handleTunnelData` for a host message iterates all `room.clients` and sends to every one. Each client receives every other client's session data. Route by session ID or use separate WS per session.

3. **TunnelStart unhandled on relay** — The client desktop app sends a `TunnelStart` message when a new TCP connection arrives, but the relay has no case for `MessageType.TunnelStart` in its switch — it falls through to `sendError(InvalidMessage)`.

4. **Predictable invite codes** — `generateInviteCode` uses `Math.random`. For a publicly-deployed relay, replace with `crypto.getRandomValues` or `crypto.randomBytes`.

5. **No message size or rate limits** — A malicious WebSocket client can send arbitrarily large payloads or flood messages, overwhelming the relay. Add `maxPayload` on the WebSocketServer and per-client rate limiting.

6. **Room timeout doesn't close WebSockets** — `cleanupExpiredRooms` deletes rooms from the map but doesn't close the host or client WebSocket connections, leaving dangling sockets that can still send messages to a now-undefined room.

7. **Reconnection produces new invite code** — When the host reconnects, `handleDisconnect` calls `createRoom()` which generates a new invite code. The client has no mechanism to discover the new code, so the tunnel is permanently broken after a transient disconnect.

## Binary/JSON protocol split

The desktop `TunnelHost` and `TunnelClient` (in `apps/desktop/src/main/`) use `encodeFrame` from `tunnel-core` to send binary WebSocket messages:

```ts
// tunnel-client.ts
const frame = encodeFrame(FRAME_TYPE_TUNNEL_DATA, sessionId, new Uint8Array(data));
this.ws.send(frame);
```

The relay handler in `relay-handler.ts` processes every message as JSON:

```ts
ws.on('message', (data) => {
  const message = JSON.parse(data.toString()) as ProtocolMessage;
  this.handleMessage(ws, message);
});
```

When a binary frame arrives, `JSON.parse` will throw, the catch block sends an "Invalid message format" error back, and the tunnel payload is dropped. The integration test avoids this problem entirely — it manually constructs JSON `TunnelData` messages with base64-encoded payloads, matching what the relay expects. This means the test proves the relay works in isolation but does not validate the desktop app's actual encoding path.

The fix is either: (a) make the relay detect binary messages and decode them with `decodeFrame`, then route accordingly, or (b) change the desktop app to use JSON-encoded `TunnelData` messages with base64 payloads like the integration test does. Option (a) is better for performance (avoids base64 expansion) but requires the relay to handle both control messages (JSON) and data messages (binary).

## Host-to-client fan-out routing

In `relay-handler.ts` `handleTunnelData`, when the sender is the host:

```ts
} else {
  // Forward from host to correct client by sessionId
  for (const client of room.clients) {
    if (client.readyState === WebSocket.OPEN) {
      client.send(JSON.stringify(message));
    }
  }
}
```

Despite the comment saying "correct client by sessionId", the code sends to *every* client. In a multi-player room, player A's session data is forwarded to players B and C. Each client would then try to look up the session ID in their local `sessions` map, fail to find it (since they didn't initiate that session), and silently discard it. So data doesn't *corrupt*, but it wastes bandwidth and leaks information between clients. In a high-throughput Minecraft game with multiple clients, this doubles or triples relay egress for no reason.

The relay would need to maintain a mapping of `sessionId -> clientWs` (populated when the client sends `TunnelStart`) to route correctly.

## TunnelStart drops silently

The desktop client sends `MessageType.TunnelStart` when a new TCP connection arrives at the local listener:

```ts
// tunnel-client.ts
this.ws.send(JSON.stringify({
  type: MessageType.TunnelStart,
  sessionId,
}));
```

The relay's switch statement has no case for `TunnelStart`, so it falls into `default: sendError(ws, ErrorCode.InvalidMessage, 'Unknown message type')`. This doesn't break the tunnel since the actual data still flows via `TunnelData` messages, but it means the relay can't build the session-to-client mapping needed for proper routing, and the client receives a spurious error on every new connection.

## Invite code entropy

`generateInviteCode` in `packages/shared/src/index.ts`:

```ts
const INVITE_CODE_CHARS = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789';
const INVITE_CODE_LENGTH = 6;

export function generateInviteCode(): string {
  let code = '';
  for (let i = 0; i < INVITE_CODE_LENGTH; i++) {
    code += INVITE_CODE_CHARS.charAt(Math.floor(Math.random() * INVITE_CODE_CHARS.length));
  }
  return code;
}
```

`Math.random` is not cryptographically secure and its output is predictable with enough samples. For a locally-shared invite code this is acceptable, but if the relay is ever publicly accessible, an attacker can enumerate codes at high speed. The relay has no rate-limiting on `JoinRoom` attempts, making brute-force trivial (36^6 / 1000 attempts per second = ~26 days at leisurely pace, much faster with parallel connections).

## Room timeout leaves connections dangling

`cleanupExpiredRooms` sets `room.status = RoomStatus.Closed` and deletes from the map, but never calls `room.hostWs.close()` or iterates `room.clients` to close them. The host and client WebSockets remain connected to the server, can still send messages, and will hit undefined behavior when `getRoom` returns `undefined` in the relay handler. The `handleTunnelData` early-returns on missing room so data is silently dropped, but the connections persist until the OS kills them or the ping/pong timeout fires.

## Reconnection invalidates invite code

When the host's WebSocket closes unexpectedly, `handleDisconnect` in `tunnel-host.ts` enters a reconnection loop that calls `this.createRoom()`. Creating a room generates a new invite code on the relay. The old invite code was deleted when the relay detected the host disconnect (it calls `roomManager.removeRoom`). The client, which only knows the original code, cannot rejoin.

For the reconnection strategy to work, the relay would need to support re-attaching a host to an existing room (perhaps via a secret room token issued at creation time), or the host would need a way to push the new code to already-connected clients (but they're also disconnected at that point since the relay closed them).

## Test coverage gaps

The unit tests cover `RoomManager` thoroughly (creation, joining, limits, cleanup, counts). The integration test validates the JSON-based relay path end-to-end with real TCP sockets and WebSockets — this is strong coverage of the "happy path" data flow.

Not tested: the desktop app's binary frame path through the relay (it cannot work), reconnection behavior, multiple simultaneous sessions in one room, large payloads or partial TCP reads, the `TunnelStart` message flow, ping/pong timeout eviction, or the port-detection logic.
