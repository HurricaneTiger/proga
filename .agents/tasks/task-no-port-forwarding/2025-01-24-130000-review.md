# Semantic Review (v1): Public relay mode selector and deployment configs

This change eliminates the port-forwarding requirement by adding a public relay server option. The desktop app's Settings page gains a three-way relay mode selector (Public / Local / Custom), deployment configs ship for Docker, Render, and Fly.io, and the README documents one-click deploy.

**Verdict**: NEEDS_CHANGES

## Issues

1. **Broken `useHealth` hook** - Still reads `settings.relayUrl` (the vestigial default) and appends `/health` to a `wss://...com/ws` URL, producing an invalid fetch target. Should use `getEffectiveRelayUrl()` with the same URL-to-health transformation as the main process handler.

2. **Dockerfile/Fly.io port env var mismatch** - Both set `PORT` but the relay server reads `RELAY_PORT`. Works by coincidence (default is 3000). Set `RELAY_PORT` instead in both configs.

3. **No custom URL validation** - The custom relay URL input accepts any string. A malformed URL causes a silent connection failure. Add basic URL validation (must start with `ws://` or `wss://`).

4. **Duplicate `PUBLIC_RELAY_URL` constant** - Declared in `packages/shared` and re-declared in `useIpc.ts`. Import the shared constant to avoid drift.

5. **Vestigial `relayUrl` field in Settings** - The Settings interface still carries `relayUrl` which is set in defaults but never written by the save handler. Remove it or repurpose it properly.
