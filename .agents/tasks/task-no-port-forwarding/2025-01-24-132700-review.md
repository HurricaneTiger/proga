# Public relay mode selector and deployment configs (v2)

This change eliminates port-forwarding requirements by adding a public relay server option. The desktop app gains a three-way relay mode selector (Public / Local / Custom) in Settings, deployment configs ship for Docker, Render, and Fly.io, and a `getEffectiveRelayUrl()` helper centralizes relay URL resolution across the renderer. The v1 review identified five issues; four are resolved and one (duplicate constant) is acceptable given the renderer/shared bundle boundary.

**Watch for:** Dead `check-relay-health` IPC handler registered but never wired through preload or called from renderer (confirmed). `useHealth` hook re-derives settings on mount only — mode changes in Settings don't propagate to already-mounted health checks until next page navigation (confirmed).

**Verdict**: APPROVED

## High-level view

`getEffectiveRelayUrl()` maps the `relayMode` enum to a concrete WebSocket URL, used by both `CreateRoomPage` and `JoinRoomPage`. The `useHealth` hook feeds the effective URL through `deriveHealthUrl()` to produce an HTTP health endpoint, but captures settings once on mount (empty deps array), creating a staleness window when settings change.

The `check-relay-health` IPC handler in the main process is dead code — never exposed in the preload bridge, never called from the renderer. The renderer's `useHealth` hook does its own direct `fetch()` instead.

## Health hook staleness

The `useHealth` hook has an empty dependency array, so it captures settings once on mount:

```typescript
useEffect(() => {
  const settings = loadSettings();
  const effectiveUrl = getEffectiveRelayUrl(settings);
  const healthUrl = deriveHealthUrl(effectiveUrl);
  // ...
}, []);
```

If a user changes relay mode on the Settings page, any other mounted component using `useHealth` will continue polling the old URL until unmounted and remounted. The health indicator lives on the main page and the user navigates away to Settings and back, so the remount usually happens — but it's a latent staleness issue that would manifest if the indicator were ever placed on the Settings page itself.

## Dead IPC handler

The main-process `check-relay-health` handler performs `url.replace(/\/ws$/, '') + '/health'` — the same transformation as the renderer's `deriveHealthUrl()`. It exists to enable health checks from the main process (bypassing CORS), but is never registered in the preload bridge. Either remove it or wire it up and have `useHealth` call it via IPC, which would also solve potential CORS issues when checking remote relay servers from the renderer's origin.

## Duplicate `PUBLIC_RELAY_URL` constant

`PUBLIC_RELAY_URL` is declared in both `packages/shared/src/index.ts` and `useIpc.ts`. The renderer never imports from the shared package — its tsconfig uses `moduleResolution: "bundler"` without workspace path mappings, making this a deliberate architecture boundary. If the values ever diverge, there's no compile-time guard, but for a single hardcoded URL the risk is low.
